Introduction

You won't blame me if I dare say that one of the most important aspects of designing a programming language is its semantics—especially the semantics of the memory model.1 The memory model tells us how the language behaves when we manipulate memory, whether manual or automatic. For instance, aliasing rules, thread visibility, ordering of memory operations (e.g., atomicity), and so on. The memory model defines what it means for a program to be safe, by specifying which behaviors are well-defined and which fall into undefined behavior (UB). However, it is the responsibility of the language's type system and static analyses to ensure that safe code cannot violate these constraints. Ideally, we would like to eliminate UB altogether, at least for a subset of the language.2
